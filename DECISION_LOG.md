# سجل القرارات التقنية/التصميمية (أثناء البناء)

هذا السجل يوثّق قرارات التنفيذ التي اتخذتها أثناء بناء الصفحة — وهو مختلف عن "قرار العميل" الذي
يظهر داخل الصفحة نفسها لكل قسم.

## القرار 1 — بنية بيانات موحّدة بدلاً من صفحات منفصلة
**السبب**: 23+ قسمًا بمحتوى غني (نصوص، جداول، مخططات) يصعب صيانته كـ JSX خام متكرر. تم فصل المحتوى
(TypeScript objects/arrays مطبوعة الأنواع) عن مكوّنات العرض، لتسهيل التعديل والتوسعة لاحقًا دون لمس
منطق الواجهة.

## القرار 2 — Tailwind CSS v4 عبر `@tailwindcss/vite`
**السبب**: تم تثبيت Tailwind v4 تلقائيًا عبر npm. تُفضَّل طريقة الإعداد الرسمية الجديدة
(`@import "tailwindcss"` + `@tailwindcss/vite` plugin) بدلاً من `tailwind.config.js` التقليدي،
لأنها الطريقة المعتمدة في v4 وتقلل الإعداد اليدوي.

## القرار 3 — لا Router
**السبب**: الموقع صفحة واحدة بالكامل (Single Page) بتمرير سلس (`scroll-behavior: smooth`) وروابط
تنقل داخلية (`#section-id`)، بدون حاجة لـ React Router، لتفادي تعقيد غير ضروري ومشاكل GitHub Pages
مع الروابط الفرعية عند تحديث الصفحة (F5).

## القرار 4 — تخزين القرارات: مفتاح واحد في localStorage بصيغة JSON مُصدَّرة بالإصدار
**السبب**: `almasa-lab-proposal:decisions:v1` يخزّن كائن واحد لكل الأقسام + الاقتراحات + الملاحظات،
لتبسيط القراءة/الكتابة/التصدير، مع لاحقة إصدار تسمح بتغيير الشكل لاحقًا دون كسر بيانات قديمة.

## القرار 5 — الأقسام بدون تفاعل تُعتبر "معتمدة كما هي" ضمنيًا
**السبب**: طلب صريح من العميل. لم يتم تخزين حالة افتراضية صريحة في localStorage لتفادي التضخم،
بل تُحسب هذا المنطق في وقت العرض/التصدير: أي قسم بلا قرار مسجّل → يُعرض ويُصدَّر كـ "معتمد كما هو".

## القرار 6 — بيانات كتالوج التحاليل والقيم المرجعية
**السبب**: طلب العميل صراحة عدم استخدام قيم طبية عشوائية من الإنترنت، وأن كتالوج التحاليل الفعلي
يجب استخراجه من قاعدة Access الحالية واعتماده من إدارة المعمل. لذلك تم عرض هذا القسم كخطة عمل ونموذج
هيكلي (اسم التحليل، الوحدة، المعدل المرجعي، السعر كأعمدة) بدون تعبئة قيم طبية حقيقية افتراضية، مع
توضيح صريح في النص أن الأرقام الظاهرة (إن وجدت) هي أمثلة توضيحية للتصميم فقط وتُستبدل من ملف Access
الفعلي.

## القرار 7 — لا Backend فعلي؛ الذكاء الاصطناعي وواتساب يُعرضان كـ Mockup تفاعلي
**السبب**: الصفحة عرض تصوري (Proposal)، ليست تطبيقًا يعمل بالفعل. لذلك مكوّنات "شرح النتيجة بالذكاء
الاصطناعي" و"إشعارات واتساب" هي محاكاة بصرية (mockup) توضيحية فقط، تُبرِز آلية العمل والقيود
(لا تشخيص، لا وصف علاج) دون استدعاء أي API حقيقي.
